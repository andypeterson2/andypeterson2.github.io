Architecture Decision Log

Six shipped. One proposed.

Status  Six accepted & live · one proposed (007) Deciders  A. Peterson (sole maintainer) Date  2026-07-08 · rev 2026-07-12 (006–007 folded in)

The subject is src/editor — a Svelte 5 island on a static Astro site (→ GitHub Pages), talking to an Access-gated Express/SQLite backend on Railway through a Cloudflare Worker gateway. The first six record decisions are shipped and live — 006, the versioned store, most recently. The last is a proposal (007) for where the data goes next. What follows is the rationale worth keeping, not every tweak.

The spine

The interface must not lie about what it can do, and a demonstration should be the real thing rather than a mock of it.

The red “disconnected” dot, the unsatisfiable sign-in prompt, the copy promising a PDF compile the demo can’t run, the inert menus styled to look live — the same bug in different clothes. Naming it made the rest fall out, and it chained the decisions: 001 created a debt, 003 paid it, and 005 couldn’t make the menu honest until 003 existed.

Constraints that shaped every call

No server of our own

All logic runs in the browser island; the only backend is the gated CV API.

The backend is owner-only

Almost every visitor reaches only the offline demo. The demo is the common case.

Offline-capable

Every feature works with no backend, and degrades to persistence when connected.

No Svelte plugin in vitest

Anything in a .svelte.ts runes module is unreachable from a unit test.

Single user

No collaboration; no concurrent editors of one document. The MCP tools are out-of-band.

The decisions are causally chained, not a menu of independent choices
001

Demo mode is the confident default, not a failed connection

Accepted · live · c3c2b3c

The editor auto-probes the backend on mount. For everyone but the owner that probe fails (Access-gated) — and the UI drew that as an error: a red dot, a filled ◆, and a “Connect to the live backend” CTA nobody outside the Access team can satisfy. The system was drawing its own happy path as a fault.

Decision

Treat demo as a first-class default — hollow ◇ status (the System-6 idiom for “unset”), a quiet invitation strip (“This is the real editor… Nothing is saved.”), sign-in demoted to a text link, and a Reset demo safety net.

Options considered

OptionComplexityHonestyPortfolio value
A · Keep the failed-connection framingLow (status quo)Actively misleadingLow — reads as broken
B · Invitation reframe  ◆ CHOSENLowHigh — states “nothing is saved”High — recruiters touch a live tool
C · Sign-in wall for non-ownersMediumHighZero — the demo is the pitch

Trade-off

C is the “safe” honest answer but destroys the reason the editor is public. A is cheap but lies. B keeps the confident default at the cost of one new obligation: if you invite people to edit, you owe them an undo — which seeded ADR-003. The residual “is this real?” risk is bought off with explicit copy, not more chrome.

  • +Red is freed for real errors; introduced semantic tokens --state-live / -busy / -error.
  • →Created the debt that forced ADR-003 (undo).
  • ⚠Revisit if the backend opens to multiple users — the “demo = default” assumption inverts.
002

The guided tour drives the real store, not a parallel renderer

Accepted · live · c8ac487

The best features are invisible — a variant lens looks like “text turned grey.” We wanted to show them. The question was what the tour actually animates.

Decision

The tour issues the same public store calls a user’s clicks make — there is no separate playback path. Its signature behaviour is the interrupt: any pointer, key, or wheel outside the tour’s own chrome yields control instantly.

Options considered

OptionFidelityDrift riskInteractivity
A · Scripted video / GIFLowAges out of sync with the appNone
B · Parallel “playback” componentMediumHigh — a second path that driftsFake
C · Drive the real store  ◆ CHOSENExactNone — it is the appReal — grab the wheel

Trade-off

B is the tempting middle ground and the wrong one: the thing you demo stops being the thing you ship. C is the most honest and makes “grab the wheel” trivial — but demands three things a mock wouldn’t:

Idempotent steps
Resume re-enters the current step, so typing a bullet reuses the same bullet, never adds a second.
A real abort
AbortSignal + a generation counter kill the in-flight typewriter and dwell timer the instant the visitor moves.
A hard write-gate
The tour mutates (it adds a bullet), so canRun() restricts it to demo mode and an $effect ends it if a session goes live mid-tour — else ?tour=1 on the owner’s session would edit a real CV.
  • +Made 001 and the tour mutually safe: demo never hits the network, so the tour can’t raise a save-error toast; the tour resets the demo first, so each step is deterministic.
  • +prefers-reduced-motion degrades to a manual Next ▸ step-through — the tour moves the page, a vestibular hazard, not a nicety.
  • ⚠Revisit if a step must demonstrate a connected-only feature — the demo-only gate blocks that by design.
003

Undo as a command stack over the reactive store

Accepted · live · 0a18590

ADR-001 promised “edit anything.” That needs an undo. The editor autosaves through a normalized REST API with optimistic mutation and temp-id → server-id reconciliation — which constrains the design far more than a local-only editor would.

Decision

A command pattern: each undoable action records { undo, redo } closures that drive the same public store methods a user’s action does. Not document snapshots.

Options considered

OptionCorrectness vs backendMemoryCost
A · Snapshot the document (memento)Fails — a restore doesn’t replay the API writes; local & server divergeGrows with doc size × depthLow locally
B · Command pattern + inverses  ◆ CHOSENPersists — the inverse issues the same PUT/POST/DELETEBounded; depth-capped at 60High — an inverse per op
C · Event sourcing / CRDTOverkill—Solves collaboration we don’t have

Trade-off — B is the only option that keeps local and server in step

A snapshot restore would silently desync the backend. The cost of B is a cluster of sub-decisions, each forced by the environment:

The field shadow
Components bind:value into state and then call saveEntry, so the old value is already gone. A WeakMap shadow of last-recorded values, diffed per save, recovers it.
Key by identity, not id
The inverse of a delete is a create, and the server issues a new id. The JS object survives; the id doesn’t. (Collides with the $state proxy trap — a pushed literal is never === the element you read back — so every insert is re-read before capture.)
Pure core + shell
Forced by the no-plugin vitest: coalescing / merge / shadow logic lives in undo.ts where tests reach it; only the runes live in undo.svelte.ts.
The applying guard
Inverses call recording methods; without a guard, undo pushes a command and the stack never drains.
Coalescing
900 ms window; merge keeps the older undo + newer redo, so undo takes back a typed word, not a keystroke.
Drop where unsound
Demo reset, variant delete, cover-letter switches drop history rather than fake it — an “Undo” that writes to a dead row is worse than a greyed-out one.
⌘Z in a field
Defers to the browser — native undo moves the caret with the text and still persists (its input event routes through saveEntry).
  • −Every future mutation must author an inverse to be undoable — a real tax, paid per feature.
  • +The pure/shell split became the template the rest of the editor already followed (slice controllers + injected host).
  • ⚠Revisit when an op has no modelable inverse — extend the drop policy, don’t fake it. (This is exactly why layout choice is deliberately not undoable: its inverse can be a null default the API can’t express.)
004

Per-profile undo scopes, with a reused (cached) working tree

Accepted · live · 6f65ba3

Undo commands close over their document’s objects. selectPerson refetched on every switch, rebuilding the whole tree — so keeping a stack across a switch would leave commands pointing at detached objects that write invisibly to the backend.

Decision

Undo is scoped per profile (stash / restore each profile’s stacks on switch), and returning to a profile reuses its cached working tree instead of refetching, so the commands’ captured objects stay valid.

Options considered

OptionPreserves historyCorrectnessSurprise
A · Clear undo on switch (status quo)No — loses work on a glanceTrivially correctLow but annoying
B · One global stack; undo auto-navigatesYesAsync load mid-undo can fail; objects still need to be aliveHigh — undo jumps profiles
C · Per-profile scopes + cached trees  ◆ CHOSENYesSound iff objects stay aliveLow — undo applies to the visible profile

Trade-off

Both B and C need the objects to survive the switch; C additionally keeps undo local to the active, visible profile. The load-bearing realization: reuse, not refetch is what makes scoping sound — a refetch rebuilds every object and strands the commands.

◆ Correction — a belief the test overturned

The first cut cached the raw fetched object. Tests proved (via a route-call counter) that no refetch happened, yet the returned profile rendered its unedited value.

The correction: Svelte’s $state does not write nested edits back through to the object it wrapped. this.person = p then editing editor.person.x mutates the proxy; the raw p stays pristine. The fix — cache this.person (the proxy), which is idempotent to re-assign.

  • −Staleness, stated plainly: a returned profile shows your working copy — an external edit (another client, the MCP tools) made while away won’t appear until reload. Acceptable for a single-user editor, and the price of not discarding your undo.
  • +Memory grows with profiles-visited-per-session — bounded and small.
  • ⚠Revisit if the editor becomes multi-user or the MCP tools become a frequent concurrent writer — then cache invalidation (or a “refresh” affordance) is needed.
005

The interface tells the truth

Accepted · live · 74f28e1 · 73312bd

A cross-cutting stance, not a feature. Several bugs this session were the same lie in different clothes — inert <span> menus styled to look clickable, a menu command that silently no-ops, hardcoded status colours, a duplicated floating-panel triad.

Decision

Three conventions, now load-bearing across the editor.

Unimplemented ⇒ disabled
Never hidden or fake. Empty menus (Edit before undo) render disabled; a command that can’t act now (Reset demo when connected) is disabled, not silently inert.
State in form, not just colour
Hollow-vs-filled glyphs, a real menuitemcheckbox with aria-checked (not a ✓ glued into a label), semantic --state-* tokens, one shared aria-live region.
Name shared primitives once
.floating-panel (toast + tour narrator) and --shadow-float, surfaced when a stylelint rule that lints CSS — but not Svelte <style> blocks — demanded a token the toast had been evading.

Trade-off

The cost is discipline — an empty menu is more code than an inert span; a real checkbox is more than a glyph. The payoff: the UI is honest to sighted and screen-reader users alike, and can’t drift into “looks live, does nothing.” Given the session began with a UI that lied, the convention pays for itself.

  • →Edit stays disabled until it has a real command — which is why undo (003) had to be built before the menu could be honest.
  • ⚠Revisit on any View / Edit growth: a new command must bring a real handler or render disabled.

The versioned store — shipped · multi-tenant — proposed

The last two look forward — at how the CV data should evolve, not the editor around it. They share their own spine, and answer the question the shipped work kept deferring: what happens when you want a history, and more than one person. 006 is now built and live, merged across two PRs and deployed; 007 remains a design.

The second spine

You don't want git. You want a slice of it — durable history and honest branches — and the constraint that keeps it cheap is single writer per document, not single user.

006

A versioned CV data store — history, without the merge

Accepted · live · 3 increments

The editor can undo an edit and show one résumé through different lenses, but it can't remember: close the tab and the history is gone. There is no “the version I sent to Google,” no diff between two drafts, no academic-vs-industry lines evolving side by side. This proposes giving the data a history — borrowing git's shape while refusing its most expensive idea.

Decision

A commit-graph-lite: immutable, content-addressed snapshots with named refs — tags (frozen) for provenance, branches (movable) for audience lines. Walk history to time-travel; compare two snapshots to diff; fork a ref to branch. No three-way merge. Ship snapshot-and-restore first.

✓ Shipped — all three increments live

1 snapshot / restore / a History drawer · 2 a structural diff (Compare) · 3 branches, tags, and cherry-restore. A HistoryController in the editor over a versions table (snapshots + a branch / parent / tag graph), reusing export/import. Merged as cv #25 + website #133 and deployed (Railway migrations 013–014 + GitHub Pages); 358 unit + 42 e2e + 335 backend green. Two connected-mode follow-ups deferred by design: persisting the current branch, and routing cherry-restore through the entry-write path.

One model — the four jobs are operations on it

◆ Not a reversal of ADR-003

ADR-003 rejected document snapshots for undo — they can't express a cheap per-field inverse, so undo stayed a command stack. Durable history has the opposite grain: whole-document states are exactly what you name, freeze, and restore. Same primitive, opposite concern — both verdicts correct.

Options considered

ApproachCost vs. today's stackCovers the four jobsBranch / mergeVerdict
Snapshot-per-commit  ◆ CHOSENLow — a versions table of JSON blobsAll fourBranch cheap; no native mergeThe core
Event-sourced log (persist undo)Medium — CRUD → append+ per-field blameSame↑ Later — enhancement
Temporal DB tablesMediumNo branch storyNoneRejected — fails job D
Dolt / git-as-storeHigh — a VC engine behind a CRUD API+ mergeFull gitRejected — pays for unused merge

The sharp calls

Snapshot, then log
Whole-document snapshots first — free at CV sizes; layer the persisted undo-log later for per-field blame + cross-session undo. Its client half already exists (ADR-003).
Branches fork, never merge
Independent after the fork; move one entry with cherry-restore by id — merge's 20% of value at 5% of its cost. Governed by: same bullet on two branches, fix one — must the other update? If yes, you've asked for merge.
Branch ⊥ variant
A branch is a line of history (time); a variant a view within one version (space). Keep them orthogonal, or you get a two-dimensional matrix nobody can hold.
Content-address the snapshot
Hash the document — free dedup, identity, and “did this change?” as a hash compare. Diff and history both stand on it.
  • +The undo stack (ADR-003) can finally persist — durable history and cross-session undo become one feature.
  • +Provenance is exact: “the CV Company X saw” is a frozen tag, reproducible forever.
  • ⚠Revisit if documents grow large — per-version snapshots stop being free; switch to deltas / the event log.
007

Multi-tenant editing with review roles

Proposed · extends 006

“What if I want merge, and multiple users?” Pulled apart: for what's actually wanted — many people each owning their own CV, and reviewers who suggest edits — you need neither merge nor a CRDT. You need auth. The constraint was never “single user”; it's single writer per document. Owners write their own, reviewers propose, nobody co-writes — so 006's store is untouched, and only who you are and what you may touch grows.

Decision

Extend 006 (do not supersede it) with three layers, settling auth as a hybrid — delegate authentication to an identity provider; own authorization, tenancy, and sharing in the app:

  1. Identity & tenancy — row-level ownership; you touch only your own documents.
  2. Per-document roles — owner / reviewer / viewer, granted by email invite.
  3. A review layer — entity-anchored comments + suggested edits (accept = a 006 cherry-restore).

authN: reuse Cloudflare Access while tenants are invite-scoped; escalate to a product auth service for open signup. The demo (ADR-001) stays public — auth gates data, never the demo.

Two stages — authenticate at the edge, authorize in the app

The auth spine — options considered

The sharp calls

Delegate authN, own authZ
Never store passwords — an IdP proves who. The app owns what you may touch: the users table, per-document roles, the sharing model. That boundary is the design.
Reviewers propose, never write
The invariant that keeps merge out. Suggestions land against entity ids; only the owner applies them — serialized through one writer, no reconciliation, ever.
Row-level ownership, server-side
“Are you the owner?” becomes “do you own — or hold a role on — this document?”, enforced on every endpoint. Never trust the client to scope its own reach.

⚠ Access is not authorization

Cloudflare Access answers who are you — not which documents you own or what a reviewer may do. That is the app's job, on every request. The login wall is not a permission system; letting it masquerade as one is the same “CORS-is-not-access-control” mistake in a new coat. authN at the edge, authZ in the app — always both.

  • +006's store is unchanged; review reuses branch + cherry-restore; the demo (ADR-001) survives multi-tenancy.
  • −The real lift is authZ — the owner-vs-demo binary becomes row-level ownership + roles on every endpoint. It reshapes the security model, not the data model.
  • ⚠The door 006 left open — if reviewers ever gain direct write, concurrent writers return and the merge / CRDT question reopens in earnest.
